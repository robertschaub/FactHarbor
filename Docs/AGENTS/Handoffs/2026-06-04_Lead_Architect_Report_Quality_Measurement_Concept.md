**Key decisions:**
- **Reuse, not reinvent.** The decomposition the request asks for already exists as the Q-code catalog: `Q-S1.*` = atomic-claim extraction (C1), `Q-EV*` = evidence (C2), `Q-V*` = per-claim verdict (C3), `Q-BE*` = overall verdict gold bands (C4), `Q-ST*` stability, `Q-HF1` runtime-integrity gate. The gap is a *rating model* + *build-comparison methodology* + an *honest reference model* on top — not new criteria.
- **Organizing spine = the reference gradient.** Gold ("right answer") exists only at the aggregate verdict (`benchmark-expectations.json`) and a few coarse floors; C1/C2/C3 detail has none, and run-to-run drift (claim-set Jaccard ≥0.6 tolerated; evidence-pool Jaccard 0.10–0.29) is the primary confound for per-component isolation.
- **Build comparison is ordinal, not score-then-subtract.** Where there's no gold, compare builds by **pairwise A/B judging** on the same input (repurpose the `/report-review` Advocate/Challenger/Reconciler panel, blind + order-balanced), aggregated to win-rates. Absolute scoring only where anchored (C4 gold bands, structural Q-codes).
- **Vector primary; scalar gates, never ranks.** Per-component finding-vector + sub-scores is authoritative; a lossy report tier (PASS/WATCH/BLOCK) only gates release readiness. Build A > B only by Pareto-dominance across the vector or an explicit Captain-owned weighting.
- **Calibration is a first-class build-layer dimension** (the plastic-en overconfidence failure mode: FALSE 9/78 *and* MOSTLY-TRUE 85/88 both overconfident), measurable only where gold exists × many reps.
- **Phase 1 = zero new spend** — roll existing Q-code findings over already-stored reports into the vector/tier and prove the comparison plumbing before any judge/live-spend. Fits the era-study Phase-3 "normalize + compare" slot.

**Warnings:**
- **Do not over-claim per-stage causal attribution across builds.** Truly isolating "build A's Stage 4 is better, Stage 1 held constant" needs cross-version claim/evidence **injection** — the deferred in-tree adapter machinery (era-study conditions ③ & ④). Without it, cross-build C3 must use semantic claim alignment (lossy) or step up to C4.
- Cross-provider judge depends on OpenAI Pro access that **may have lapsed (~2026-05-26)** — treat as aspirational; the built-in multi-vote debate panel is the available fallback.
- A judge must be **anchored on gold-band families first** (does its A/B ordering agree with band agreement?) before being trusted on ungolded sub-stages — guards against self-grading circularity.
- This is a **concept only**. No code, no schema changes. Implementation deferred to Captain sign-off.
