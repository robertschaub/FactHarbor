**Root cause:**
Hidden jobs were invisible to the internal runner because runner reads used `GET /v1/jobs...` without `X-Admin-Key`. The API correctly hides hidden jobs from non-admin reads, so the runner received `404 {"error":"Job not found"}` for real hidden jobs and marked them failed. Runner writes already used the admin key through `apiPutInternal(...)`; reads were the missing half.

**Production evidence gathered:**
- `https://app.factharbor.ch/api/health` was healthy; `https://app.factharbor.ch/api/fh/health` reported DB connectivity.
- Deployed API SHA during diagnosis: `2f7a2805c44c3a72f9102c94f54aee30bd114ba9`.
- Admin-visible jobs: 189 total, 176 `SUCCEEDED`, 6 `FAILED`, 6 `INTERRUPTED`, 1 `RUNNING`.
- Every non-successful job was hidden.
- Every failed job had the same event sequence: `Job created` -> `Triggering runner` -> `Report hidden by admin` -> `Runner started` -> `API GET failed 404: {"error":"Job not found"}`.
- No raw user inputs, reports, credentials, or secrets were copied into this handoff.

**Code change:**
`apiGet(...)` in `internal-runner-queue.ts` now accepts an optional `adminKey` and sends `X-Admin-Key` when present. All internal runner reads now pass the existing `adminKey`:
- initial job read in `runJobBackground`
- final status guard read before marking `SUCCEEDED`
- paginated stale/recovery list read
- live orphan refresh read
- queued job detail read before background launch

**Regression test:**
Added a test that simulates hidden-job reads returning 404 unless the admin header is present. The test verifies that the hidden queued job reaches `runClaimBoundaryAnalysis`, emits `RUNNING`, and all internal GET reads include `x-admin-key`.
