**Key decisions (all Captain-confirmed 2026-06-04):**
- **In-tree variant registry**, not process-per-variant. Two halves: declarative `apps/web/configs/variants.json` (read by BOTH the C# API allowlist and the web dispatcher) + code binding `apps/web/src/lib/analyzer/variants/registry.ts` (`id → entry fn`). Stable entry contract `(ctx: PipelineRunContext) => Promise<PipelineRunResult>`. Add = module + 1 binding + 1 manifest row + prompt profile + optional override file; no orchestrator/stage edits. This **revives the dormant `pipelineVariant` seam** (already persisted on `JobEntity` at creation) + the historical single-dispatcher pattern. The May-12 `analyzer-v2` rebuild becomes just one more registry entry.
- **UCM = files authoritative; layered base + sparse per-variant override.** Base = existing `apps/web/configs/{pipeline,search,calculation,sr}.default.json` read directly at runtime. Per-variant `configs/variants/<id>.override.json` overrides a subset of keys (e.g. `llmProvider`, `modelVerdict`). Effective = `deepMerge(base, override)` (objects merge by key; arrays+scalars REPLACE) then Zod-validate. Sharing principle = control-variable vs independent-variable line; every override is a recorded confound.
- **Prompt sharing — settled by a FULL `/debate` (2026-06-04, verdict MODIFY; Arch §4.3a, Plan Phase 3b).** The "should prompts also use base+override like config?" question was reopened and adversarially reviewed (Advocate/Challenger/2 Probes/Reconciler/Validator; both probes STABLE=NO; confidence SPECULATIVE; Validator PASS-WITH-CAVEATS). Outcome: **Phase 1 ships Option A (wholesale per-variant prompts); Option B (section-level base+override, base-optional) is CONDITIONAL Phase 3b** — built only if a pinned G1 variant-population table shows genuine section-level divergence, and only behind a **section-presence/completeness validator**. Key finding: B's coherence gap is a *silent missing-section* hole (a variant dropping e.g. the neutrality/language directive still renders valid schema) that whole-section-replace + render/schema tests do NOT catch. A-first rationale: A = B-with-no-base, so deferring B forecloses nothing and costs no rework; building B's merge machinery pre-G1 is machinery-first risk (Pipeline_V2 lesson).
- **Remove** the config DB (`config_blobs`/`config_active`/`config_usage`), versioning/activation/rollback/history, and all editing UI. **Keep** a read-only inspector (effective config + overrides + prompt + fingerprint per report/job). Keep `DEFAULT_*_CONFIG` + `config-drift.test.ts` as TS fallback + parity guard.
- **Provenance:** each report embeds `meta.provenance` (variantId, code commit, per-prompt-file hashes, base/override/effective config hashes + overriddenKeys, embedded effective config, model map) + a **pipeline fingerprint** = `sha256(variantId + commit + prompt hashes + effective config hashes + models)`. Commit hash ALONE is insufficient. Denormalize `PipelineFingerprint`/`ConfigEffectiveHash` onto `JobEntity`. **Config is embedded; prompt is by-reference (hash+git)** — deliberate asymmetry (212KB prompt would bloat ResultJson); full prompt reconstruction guaranteed only under commit-first discipline.
- **Parallel = both:** per-job selection first (foundation), then a matrix run (one input → N variants sharing `MatrixRunId`/`InputGroupId` → comparison view with provenance diff).
- **Rebuild-old-variant workflow:** new variant id + re-implemented module + prompt copied from commit X + override = `X.effective − current.base` (computed from X's embedded effective config, since base may have drifted). "Does the same" = behavioral parity within bands, not bit-exact.
